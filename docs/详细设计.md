# 系统详细设计

本文档涵盖了目前系统已规划和实现的核心业务模块的详细设计。相关数据库建表脚本统一维护在 `docs/项目所有表结构.sql` 中。

---

## 模块一：用户管理模块

### 1. 需求概述
- 提供系统基础的访问控制与多用户隔离机制。
- 确保只有处于登录状态的用户才能访问业务系统，并强制要求各个业务数据模块按登录用户的 ID 进行数据隔离。

### 2. 数据库设计方案
- 涉及核心表：**`sys_user`（系统用户表）**
- 字段包含：主键 `id`、用户名 `username`（系统内唯一）、密码 `password`、帐号状态 `status`（1正常，0禁用）。

### 3. 核心功能与安全交互逻辑
1. **用户认证与状态保持**：
   - 接收前端传入的用户名和密码，利用基础设施层的 Repository 模块查询匹配。
   - 验证通过后，将用户信息（如 `user_id`）写入服务端 Session 以维持登录状态。
2. **全局安全拦截（LoginFilter）**：
   - 拦截除登录接口、静态资源白名单（如 `.css`, `.js`, `.png` 等静态文件扩展名）外的所有请求。对于未登录请求予以重定向或拦截响应。
   - **防路径穿越机制**：对请求 URI 进行严格的规范化处理（Canonical Path Validation），识别并拦截包含 `..`, `//`, `;` 等异常字符的路径，杜绝恶意读取或越权访问服务器内部资源。
3. **架构隔离设计（Repository 模式）**：
   - 在 `cyc-infra` 层定义 `SysUserRepository` 接口并提供实现，彻底隔离 MyBatis 的 `SysUserMapper` 组件，禁止 `cyc-service` 业务逻辑层直接与底层 SQL 映射器耦合。

## 模块二：投资记录管理模块

### 1. 需求概述
旨在提供一个轻量级的**个人投资记录管理系统**，满足以下核心需求：
- **资产分类管理**：支持将投资产品分类（如基金、股票、ETF、固定收益类）。
- **资金流水记录**：追踪具体的投资行为（买入、追加投入、卖出等），并详细记录每次的金额、单价、份额与时间。
- **汇率与货币管理**：记录跨币种投资产生的换汇流水（支持 CNY/HKD/USD/USDT 等）。纯记录性质，无严格账户余额约束。
- **收益追踪**：支持记录与查看特定产品的累计收益和单日收益。收益由用户手动录入，前端支持联动计算。
- **多用户强隔离**：所有投资及流水数据均绑定 `user_id`，杜绝越权访问他人账单。

### 2. 数据库设计方案
涉及 4 张核心业务表，字段均包含 `user_id` 以作鉴权隔离：
1. **`inv_product`（投资标的表）**：记录产品名称、代码、分类、计价币种及状态。
   - **设计细节**：`type`（分类）字段采用 `VARCHAR(20)` 类型，直接映射后端的 `ProductTypeEnum` 枚举（如 `FUND`, `STOCK`），避免魔数，便于代码统一维护。
2. **`inv_record`（交易流水表）**：记录每一笔买入/卖出的金额、单价、份额及手续费等。
   - **设计细节**：引入 `value_date` (起息日/确认日) 字段，支持“T+N 确认机制”。分离交易扣款时间与起息时间，用以识别“在途资金”。
3. **`inv_exchange`（换汇记录表）**：跨币种划转纯流水账记录。
   - **设计细节**：支持**双向汇率存储**，包含 `exchange_rate` (买入/卖出) 和 `exchange_rate_reverse` (卖出/买入) 两个字段。
4. **`inv_profit_snapshot`（收益快照表）**：记录用户手动录入或前端计算的累计收益与单日收益。

### 3. 核心功能与前端交互逻辑
1. **标的管理模块**
   - 新增/编辑产品信息，支持数据回显。
   - 列表查询、分类筛选以及产品删除功能（带后端数据归属鉴权防覆盖机制）。
2. **交易流水模块**
   - 针对某个特定标的，录入买入或卖出记录，**同时支持流水的修改与删除操作**。
   - **在途资金与起息控制**：支持灵活配置起息日（默认同交易日），前端双行并列展示。用户在录入收益快照前，可藉此清晰分辨资金是否处于“在途”状态，进而精准剥离未起息资金，防止收益率计算失真。
   - **防空覆盖安全机制**：在执行流水修改（Update）时，后端服务层自动核对权限，并强制继承原有的 `user_id`、`product_id` 和 `create_time` 等底层核心元数据，杜绝更新过程中发生的字段覆盖或丢失风险。
3. **换汇管理模块**
   - 独立录入换汇记录，仅作流水账功能。
   - 支持编辑与删除流水的双向操作。
   - **双向汇率自动推演与展示**：前端动态监听“买入金额”和“卖出金额”，实时自动计算出**“买入/卖出”**与**“卖出/买入”**双向汇率并填入表单；列表页也同步直观地展示该双向汇率数据。
   - **多币种流转动态汇总**：通过前端轻量级计算实现各币种资金的“流入（+）”与“流出（-）”统计，利用金融红绿配色方案及极简符号直观呈现在流水账面板顶部，在零服务器开销的前提下提供全局资金总览。
   - **附加信息记录**：全面启用并打通底层已设计好的 `remark` 字段，允许在前端录入和列表视图中直观地补充、查看换汇相关的附加信息（如地点、用途等）。
4. 收益视图与录入交互（非连续记账处理）
   - **前端自动计算逻辑（基于 T-1 数据）**：
     - 当用户录入“累计收益”时：系统查找**前一天（T-1）的快照值**，自动计算并带出“当日收益”（当次累计收益 - 前一天累计收益）。
     - 当用户录入“当日收益”时：系统查找**前一天（T-1）的快照值**，自动计算并带出“累计收益”（前一天累计收益 + 当日收益）。
     - **断层处理**：系统会动态比对当前录入日期是否恰好为上一笔历史快照日期（新增模式下为最新一笔，编辑模式下为排序后的前一笔）的**后一天**。若不满足恰好相差一天（连续记账）的校验，系统将**不自动计算/推演**当日收益与累计收益的差额，允许用户同时手动填入这两个数据，防止不连续日期下的收益计算失真。
    - **批量快速录入与变动高亮机制**：支持对选定产品在同一弹窗中进行批量快速录入。为了极致优化输入体验，输入框保持空白，免去用户每次退格清空旧值的繁琐操作；同时，每个输入框右侧附加了小巧的“旧: XX”字样的输入组标签（Input Group Text）以直观呈示上期快照值。前端动态监听输入事件并比对输入值与该卡片所绑定的初始快照备份值的差异：默认展示灰色的“不更新”状态标签，一旦检测到任意字段发生了与历史值的实质性改变，卡片边框将自动亮起绿色高亮，并切换展示绿色的“待更新”状态标签。在输入联动过程中，系统会动态校验当前选定的“新快照日期”是否恰好为“上期历史快照日期”的后一天：**仅在日期恰好相差一天（连续记账）时，才触发“累计收益”与“当日收益”的双向自动联动**，以防在不连续的记账断点处发生收益误算。在点击“确认录入”提交时，未填写的字段（保留为空白）会自动智能继承其历史快照的对应值进行记录。
   - **入库规则**：用户确认填写/计算的数据无误后提交，系统将 `total_profit` 和 `daily_profit` 共同落库。
   - **数据修改与保护**：支持对历史收益快照进行二次编辑与删除。在修改时，后端启用“防空覆盖”机制，自动继承原有权属和创建时间，确保底层核心数据不被破坏。
5. **数据可视化分析与偏好模块**
   - **统计图表展示**：在前端引入图表库（如 ECharts 等）。
   - **收益折线图**：展示特定产品的“累计收益走势”与“单日收益走势”。
   - **动态年化收益分析图（按需加载）**：在“交易流水”标签页后新开独立的“年化分析”标签页。支持自由切换以下两种统计周期：
     - **周年化走势**：向前追溯计算过去 12 个滚动周期（每 7 天为一周期）的阶段年化收益率，生成 12 个数据点的折线走势。
     - **月年化走势**：向前追溯计算过去 12 个滚动周期（每自然月为一周期）的阶段年化收益率，生成 12 个数据点的折线走势。
   - **每日万份收益走势**：在产品详情页新增“万份收益”标签页，严格按最新快照日期向前推算连续 30 个自然日，绘制每日万份收益柱状图。结合底槽阴影、针对性打叉（`✖`）标记以及极简的 X 轴排版，彻底解决日期断层与 0/null 视觉难辨的体验痛点。
   - **全局纵览与多产品对比**：在侧边栏提供独立的“纵览分析”视图，支持通过复选框随意勾选多个产品，并在同屏内直接横向比对它们的“累计收益走势”以及“万份收益走势”。
   - **用户级偏好持久化存储**：系统引入轻量级的用户偏好配置功能，实现“千人千面”的操作记忆。如在全局纵览中勾选对比的产品状态将自动异步提交至后台偏好库并绑定用户，刷新页面或换机登录后可无缝自动回显。
6. **系统 UI/UX 全局优化**
    - 全站网页接入自定义 `favicon.ico`，提升品牌感和浏览器标签页辨识度。
    - 重构了表单更新交互（Update 接口），实施严格的**防空覆盖保护**，防止关键元数据（如 `user_id` 和 `create_time`）在用户只修改部分字段时被意外清空。
    - **重写全局原生 `window.alert` 并接入非阻塞 Toast**：为了解决原生浏览器 `alert` 的同步阻塞导致的标签页锁死、卡顿以及样式简陋问题，全站重构重写了 `window.alert` 对象，将其无缝平替为自定义的非阻塞式右侧滑入悬浮通知（showToast）。系统会自动识别警告文案中的情绪关键词（如“成功/🎉”、“失败/❌/错误”），自动匹配渲染对应的清雅配色方案（绿/红/黄/蓝）并实现 4 秒后自动淡出消失。
